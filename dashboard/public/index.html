❓ Signal Commands Reference
CommandWhat happens
hello / hi / any questionCommander routes to the right crew (research, coding, writing) or answers directly. Short replies are interpreted in context of your conversation history.
📎 Send file(s)Attach PDF, DOCX, XLSX, JPG, or PNG to any message. The system extracts text/data from attachments and includes it as context for the request. Works with all crews. Send files alone to get a summary, or with a message like "summarize this PDF".
statusReturns system health and count of pending improvement proposals.
learn <topic>Adds topic to the learning queue. At 3 AM daily (or on manual trigger), the self-improvement crew researches it and saves a skill file to workspace/skills/.
watch <youtube_url>Extracts the video transcript, distills key knowledge into a skill file, and stores a summary in shared team memory. The learned content improves all future agent responses.
show learning queueDisplays all pending topics in the learning queue.
run self-improvement nowImmediately processes the learning queue (up to 3 topics). Agents research, extract knowledge, and save skill files.
improveTriggers an improvement scan. An analyst agent reviews current skills, tools, and crews, identifies capability gaps, and creates 1-3 structured improvement proposals.
proposalsLists all pending improvement proposals with their IDs, types, and titles.
approve <id>Approves and applies a proposal. Skill proposals copy files to skills/ immediately. Code proposals are staged to applied_code/ and activate on next container rebuild.
reject <id>Rejects and archives a proposal. No files are changed.
evolveTriggers one evolution cycle immediately. The system analyzes its own metrics (errors, skills, experiment history), proposes ONE improvement, and applies it (skills) or creates a proposal (code).
experimentsShows the experiment journal: what the evolution engine has tried, what was kept, and what was discarded.
errorsShows the 5 most recent errors with diagnosis status and recurring patterns. The self-healing system automatically diagnoses errors and creates fixes in the background.
llmShows current LLM configuration: whether using local Ollama or cloud Claude, which model, vetting status, and cost breakdown.
auditTriggers an immediate code audit. The auditor reads all Python source files, identifies bugs/security issues/anti-patterns, and auto-deploys fixes. Runs automatically every 4 hours.
fix errorsTriggers the error resolution loop. Scans for recurring error patterns, generates targeted fixes, applies them, and tracks whether they resolve the error. Loops up to 3 attempts per pattern. Runs automatically every 30 minutes.
auditorShows audit activity summary: recent code audits, error resolution status (resolved/in-progress/failed), and recent auto-deployments.
deploysShows the auto-deployment log: what files were deployed, when, and why.
fleetShows the LLM fleet status: running containers, available models in catalog, and benchmark scores. Each model runs in its own Docker container, spawned on demand and killed when idle (>10min).
fleet stop allStops all LLM fleet containers immediately, freeing all GPU/RAM.
fleet pull <model>Pulls a new model from the Ollama registry into the shared volume. Example: fleet pull gemma3:27b. The model persists and doesn't need re-downloading.
How routing works
Commander classifies every message and dispatches to specialist crews:
• Research — web search, fact-finding, current events (can spawn parallel sub-agents for complex topics)
• Coding — writes, tests, and debugs code in a Docker sandbox
• Writing — summaries, documentation, emails, reports
• Direct — simple questions answered by Commander itself
Complex requests can dispatch multiple crews in parallel (e.g. research + writing).
LLM Fleet (containerized, dynamic model selection)
Each LLM model runs in its own Docker container, spawned on-demand and killed when idle (>10min). All containers share a persistent volume so models are downloaded once.
Commander → Claude API (routing, max intelligence)

Self-improvement flow
learn topic → queued → cron or manual trigger → agent researches 3+ sources → saves skill .md file → skill loaded into all future requests
watch youtube_url → extracts transcript → agent distills key knowledge → saves skill file + stores in team memory → improves future responses
improve → analyst scans capabilities → creates proposals → you approve/reject via Signal → approved changes applied
Self-healing
When any crew fails, the system automatically: logs the error with context → spawns a background "System Doctor" agent → diagnoses the root cause → either saves a knowledge fix (immediate) or creates a code fix proposal (needs approval). Use errors to check the error journal.
Evolution loop (autoresearch-inspired)
Every 6 hours (configurable via EVOLUTION_CRON), the system runs an autonomous improvement cycle: gathers metrics (errors, skills, experiment history) → identifies the highest-impact improvement → executes ONE change (skill fix or code proposal) → logs the experiment → keeps improvements, discards regressions. Use evolve to trigger manually, experiments to see history.
Code Auditor (continuous quality)
Every 4 hours: reads all Python source files → deepseek-r1 analyzes for bugs, security issues, anti-patterns → writes fixes to applied_code/ → auto-deployer validates syntax → backs up originals → deploys fixes live. Use audit to trigger manually, auditor for status.
Error Resolution Loop (persistent fix)
Every 30 minutes: scans error journal for recurring patterns (2+ occurrences) → generates targeted code fix → applies fix → monitors if error recurs → if yes: tries different approach (up to 3 attempts) → if no: marks as resolved. Use fix errors to trigger manually, auditor for resolution status.
Auto-Deployer
When auditor or error loop writes fixes to applied_code/: validates Python syntax → backs up originals → copies to live /app/ → logs deployment. Use deploys to see history. All deploys are backed up for rollback.
