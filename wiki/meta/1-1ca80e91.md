**1. Topic:**  
Ensuring Reliable Database Transactions via ACID Properties

**2. When to Use:**  
Apply when designing or reviewing systems that require strict data integrity—such as financial operations, inventory management, or user account updates—where any partial or corrupted transaction could lead to inconsistency, loss, or duplicate data.

**3. Procedure:**  
- **Define Transaction Boundaries:** Clearly group related database operations (e.g., debit & credit) into a single logical unit.  
- **Enable Isolation:** Configure the database’s isolation level (e.g., `SERIALIZABLE` or `REPEATABLE READ`) to prevent concurrent transactions from interfering.  
- **Apply Atomic Commits:** Use transaction control commands (`BEGIN`, `COMMIT`, `ROLLBACK`) so all operations succeed or fail together.  
- **Verify Consistency:** Ensure preconditions (e.g., sufficient balance) and post-conditions (e.g., balanced ledger) are validated within the transaction.

**4. Pitfalls:**  
- **Assuming Defaults Are Enough:** Databases may default to lower isolation levels; explicitly configure based on risk tolerance.  
- **Ignoring Partial Failures:** Even with ACID, application-level errors (e.g., network timeout after `COMMIT`) require idempotency or compensation logic.  
- **Overlooking Performance:** High isolation (e.g., `SERIALIZABLE`) can cause locking/contention; balance integrity with throughput needs.  
- **Misapplying to Non-DB Operations:** ACID governs database state only—external actions (e.g., sending emails) need separate orchestration (e.g., Saga pattern).
