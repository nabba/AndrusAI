**Topic:**  
API Rate Limiting Implementation  

**When to Use:**  
- Protecting endpoints from excessive traffic or abuse  
- Ensuring service stability under high load or malicious attacks  
- Enforcing fair usage policies for clients or APIs  

**Procedure:**  
1. **Select Algorithm:** Choose a rate-limiting strategy (e.g., Fixed Window Counter, Token Bucket) based on precision and resource needs.  
2. **Define Limits:** Set request quotas (e.g., X requests/minute) and optional burst allowances per client identifier (API key/IP).  
3. **Implement Tracking:** Use a fast, shared datastore (e.g., Redis) to count requests per identifier within time windows.  
4. **Add Middleware:** Integrate logic early in the request pipeline to check counts and reject excess requests with appropriate HTTP status (e.g., 429 Too Many Requests).  
5. **Monitor & Tune:** Log rate-limit events and adjust limits/algorithms based on traffic patterns and system performance.  

**Pitfalls:**  
- **Burst Handling:** Overly permissive burst allowances can still overwhelm downstream services; tune carefully.  
- **Distributed Systems:** Ensure the datastore (e.g., Redis) is highly available; a single point of failure can block all traffic.  
- **Identifier Choice:** Relying solely on IPs may penalize shared networks; combine with API keys or user IDs where possible.  
- **Complexity vs. Need:** Advanced algorithms (e.g., sliding logs) add overhead—assess if simpler methods suffice.  
- **Client Experience:** Sudden limit enforcement can break legitimate clients; provide clear headers (`X-RateLimit-*`) and gradual ramp-up periods.
