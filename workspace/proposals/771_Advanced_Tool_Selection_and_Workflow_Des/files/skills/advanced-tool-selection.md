# Advanced Tool Selection and Workflow Design\n\n## Principles\n1. **Task Decomposition**: Break complex tasks into subtasks that match specific tool capabilities.\n2. **Tool Matching**: Choose tools based on data format, need for interaction, and expected outcome. \n   - Use `web_fetch` for static pages; use `browser_fetch` for SPAs.\n   - Use `team_memory_store` to preserve intermediate results.\n3. **Fallback Strategy**: If a tool fails, have alternative approaches (e.g., try a different selector, switch to screenshot for manual inspection).\n4. **Iterative Validation**: After each step, verify results before proceeding.\n\n## Common Workflows\n- **Research -> Synthesis**: Use `web_search` then `web_fetch`, store findings in `team_memory`, later `team_memory_retrieve` for writing.\n- **Data Analysis**: Use `file_manager` to load data, `code_executor` to process, store results with `team_memory_store`.\n\n## Decision Tree\n```\nIs task about retrieving info from web?\n├─ Yes: Is page dynamic? → Yes: use browser_fetch; No: use web_fetch\n├─ No: Is it about code? → Use code_executor\n...\n```\n\n## Crew Coordination\n- Write results to shared memory with clear metadata (crew, task_id) for other crews.\n- Use `team_memory_retrieve` to locate prior relevant messages.\n
